# Метрики И Модели В ALERTA 2.0

Этот документ описывает:
- какие данные сохраняются по каждой статье
- как формируется модель объекта
- как формируется модель угрозы
- что означают метрики и как они интерпретируются

Документ отражает текущую реализацию backend.

## 1. Общая идея

В системе каждая запись в базе представляет собой не просто новость, а нормализованный объект наблюдения.

Он состоит из трех уровней:

1. Базовый объект статьи
2. Классификационная модель
3. Модель угрозы и метрики риска

То есть на входе у нас HTML-страница источника, а на выходе:
- текст статьи
- тип материала: `news` или `threat`
- категория и подкатегория угрозы
- severity
- набор нормализованных метрик в диапазоне `0..1`

## 2. Модель объекта

Модель объекта хранится в MongoDB в сущности `Article`.

Основные поля:

- `url` — уникальный URL статьи
- `source` — название источника
- `title` — заголовок
- `text` — извлеченный текст статьи
- `publishedAt` — дата публикации
- `createdAt` / `updatedAt` — служебные timestamps MongoDB

Это базовый уровень объекта, который отвечает на вопрос:

`Что это за материал, откуда он пришел и какой у него исходный контент?`

## 3. Классификационная модель

После извлечения текста система относит материал к одному из двух типов:

- `news`
- `threat`

Дополнительные поля классификации:

- `type`
- `category`
- `subcategory`
- `severity`
- `country`
- `vulnerability_type`
- `classification_reasoning`

### 3.1 Тип материала

`type`:
- `news` — информационный, аналитический, рыночный, регуляторный или комментарийный материал без подтвержденного инцидента
- `threat` — материал, где описан конкретный инцидент, атака, эксплуатация, утечка, вредоносная активность или подтвержденная угроза

### 3.2 Категории угроз

Поддерживаемые категории:

- `Malware`
- `Phishing`
- `Vulnerabilities & Exploits`
- `Data Breach`
- `Network Attacks`
- `Supply Chain`
- `APT`
- `Fraud`
- `Cloud Security`
- `Physical-Cyber/ICS`

### 3.3 Подкатегории

Примеры подкатегорий:

- `Malware` -> `ransomware`, `spyware`, `trojan`, `botnet`
- `Phishing` -> `phishing`, `spear-phishing`, `smishing`, `vishing`, `impersonation`
- `Vulnerabilities & Exploits` -> `zero-day`, `CVE disclosure`, `exploit in the wild`, `misconfiguration`
- `Data Breach` -> `leaks`, `dumps`, `credential exposure`, `insider leaks`
- `Network Attacks` -> `DDoS`, `BGP hijacking`, `DNS attacks`, `scanning campaigns`
- `Supply Chain` -> `compromised dependencies`, `poisoned updates`, `third-party breach`
- `APT` -> `espionage`, `sabotage`, `influence operations`, `cyber warfare`
- `Fraud` -> `payment fraud`, `crypto scams`, `account takeover`
- `Cloud Security` -> `IAM abuse`, `cloud misconfig`, `token leakage`
- `Physical-Cyber/ICS` -> `SCADA`, `critical infrastructure`, `industrial incidents`

### 3.4 Severity

`severity`:
- `high`
- `medium`
- `low`
- `null`

Для `news` severity обнуляется.

### 3.5 Страна

`country` — страна жертвы, инцидента или основного контекста.

Может содержать:
- `Russia`
- `United States`
- `Global`
- и другие нормализованные названия стран

### 3.6 Тип уязвимости

`vulnerability_type` используется только для `Vulnerabilities & Exploits`:

- `0-day`
- `n-day`
- `null`

### 3.7 Обоснование

`classification_reasoning` — короткое текстовое объяснение, почему статья была отнесена к конкретному типу и категории.

Это поле полезно для:
- аудита качества классификации
- ручной проверки
- объяснимости системы

## 4. Модель угрозы

Модель угрозы описывает не сам текст, а предполагаемый профиль угрозы, связанный с материалом.

Она включает:
- объект воздействия
- характеристики атаки
- параметры эксплуатации
- возможный ущерб
- зрелость/актуальность угрозы

Эта модель представлена такими полями:

- `target_sector`
- `sub_sector`
- `attack_scale`
- `region`
- `attack_vector`
- `exposure_required`
- `user_interaction`
- `complexity`
- `exploit_available`
- `privileges_required`
- `impact_confidentiality`
- `impact_integrity`
- `impact_availability`
- `active_exploitation`
- `time_to_exploit`
- `llm_confidence`
- `extracted_at`

## 5. Модель объекта воздействия

Под "моделью объекта" в контексте угрозы система фактически описывает:
- по кому бьют
- в какой среде
- насколько масштабно

Ключевые поля:

- `target_sector`
- `sub_sector`
- `region`

### 5.1 target_sector

Нормализованные значения:

- `energy`
- `finance`
- `gov`
- `generic`

Логика:
- если текст указывает на банки, платежи, финансовые сервисы -> `finance`
- если есть госорганы, министерства, реестры -> `gov`
- если есть энергетика, power/grid/oil/gas -> `energy`
- иначе -> `generic`

### 5.2 sub_sector

Более точный подтип сектора.

Сейчас это поле предусмотрено схемой и может использоваться, но часто остается `null`.

### 5.3 region

Географический контекст воздействия.

Обычно совпадает с `country`, но используется именно как поле модели угрозы.

## 6. Метрики угрозы

Ниже перечислены все численные и булевы метрики, которые формируют профиль угрозы.

### 6.1 attack_scale

Диапазон: `0..1`

Смысл:
- насколько велик размах атаки или инцидента

Типовая интерпретация:
- `0` — угрозы нет или это просто новость без инцидента
- `0.2-0.4` — локальный или ограниченный инцидент
- `0.5-0.7` — заметная кампания
- `0.8-1.0` — массовая или критическая активность

### 6.2 attack_vector

Нормализованные значения:

- `network`
- `local`
- `adjacent`
- `physical`

Смысл:
- каким путем реализуется воздействие

### 6.3 exposure_required

Булево поле.

Смысл:
- нужна ли доступность цели для атаки или эксплуатации

### 6.4 user_interaction

Булево поле.

Смысл:
- требуется ли действие пользователя

Примеры:
- открыть письмо
- кликнуть по ссылке
- установить приложение

### 6.5 complexity

Диапазон: `0..1`

Смысл:
- насколько сложна реализация атаки

Условная интерпретация:
- ближе к `0` — простая эксплуатация
- ближе к `1` — сложная операция

### 6.6 exploit_available

Диапазон: `0..1`

Смысл:
- насколько доступен эксплойт или возможность эксплуатации

Текущая логика нормализации:
- `0.0` — ничего не известно
- около `0.65` — есть `PoC`
- около `0.9` — `active exploitation` / `in the wild`
- `1.0` — массовая эксплуатация

### 6.7 privileges_required

Диапазон: `0..1`

Смысл:
- какие привилегии нужны атакующему

Текущая интерпретация:
- `0.9` — привилегии не нужны
- `0.55` — нужен уровень пользователя
- `0.25` — нужен `admin/root`

Важно:
- здесь большее число означает меньший барьер для атакующего

### 6.8 impact_confidentiality

Диапазон: `0..1`

Смысл:
- потенциальный или подтвержденный ущерб конфиденциальности

Примеры:
- утечка данных
- компрометация почты
- кража учетных данных

### 6.9 impact_integrity

Диапазон: `0..1`

Смысл:
- ущерб целостности

Примеры:
- подмена данных
- несанкционированное изменение

### 6.10 impact_availability

Диапазон: `0..1`

Смысл:
- ущерб доступности

Примеры:
- DDoS
- ransomware
- отказ сервисов

### 6.11 active_exploitation

Булево поле.

Смысл:
- идет ли активная эксплуатация в реальном мире

### 6.12 time_to_exploit

Диапазон: `0..1`

Смысл:
- насколько быстро угроза может быть реализована

Практически:
- низкое значение — сложная и долгая реализация
- высокое значение — быстрая эксплуатация

### 6.13 llm_confidence

Диапазон: `0..1`

Смысл:
- уверенность модели в извлеченных метриках

Важно:
- в текущем стабильном офлайн-режиме это значение часто `0` для `news`
- при fallback-эвристике может использоваться техническое значение по умолчанию

### 6.14 extracted_at

Дата и время извлечения метрик.

Нужно для:
- аудита
- переоценки качества
- отладки пайплайна

## 7. Принцип нормализации

Система старается приводить данные к нормализованному виду:

- числа -> `0..1`
- булевы значения -> `true/false`
- секторы -> фиксированный набор
- векторы атаки -> фиксированный набор
- категории и подкатегории -> фиксированный словарь

Это нужно, чтобы:
- строить аналитику
- фильтровать статьи
- агрегировать угрозы по источникам и типам
- в будущем строить скоринг и визуализацию

## 8. Как формируется тип `news` / `threat`

Ключевой принцип:

- `threat` ставится только когда в тексте есть конкретный инцидент, атака, эксплуатация, утечка, вредоносная операция или подтвержденная злоупотребляющая активность
- `news` ставится, когда материал относится к общему ИБ-контексту, но без подтвержденного инцидента

Примеры `news`:
- рынок ИБ
- комментарии компаний
- кадровые перестановки
- регулирование
- исследования без инцидента
- патч без факта эксплуатации

Примеры `threat`:
- атака на инфраструктуру
- утечка данных
- эксплуатация `CVE`
- фишинговая кампания
- ransomware-инцидент
- supply-chain compromise

## 9. Эвристики, влияющие на качество

В текущей реализации важную роль играют эвристики:

- список ключевых признаков ИБ-релевантности
- правила отнесения к `news` и `threat`
- сопоставление категорий по ключевым словам
- правила расчета severity
- fallback-логика для метрик

Это означает:
- система уже полезна для поточного сбора
- но некоторые ошибки классификации возможны
- особенно на смешанных источниках и текстах без явного инцидента

## 10. Практическая интерпретация записи

Пример мысленного чтения одной записи:

- `type=threat`
- `category=Supply Chain`
- `severity=high`
- `target_sector=generic`
- `attack_scale=0.24`
- `impact_confidentiality=0.22`
- `impact_integrity=0.18`
- `impact_availability=0.2`

Это означает:
- система увидела реальный инцидент
- отнесла его к цепочке поставок
- оценила как высокий по severity
- но численные метрики пока умеренные и отражают базовый профиль угрозы, а не максимально катастрофический сценарий

## 11. Ограничения текущей версии

- часть метрик строится эвристически
- на некоторых источниках есть смешение ИБ-контента и общего техконтента
- англоязычные источники могут требовать отдельных parser rules
- без внешнего LLM часть полей формируется только эвристиками
- `sub_sector` пока используется ограниченно
