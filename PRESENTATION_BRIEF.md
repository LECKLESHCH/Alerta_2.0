# ALERTA 2.0

## Краткое описание проекта

**ALERTA 2.0** — это аналитическая система мониторинга киберугроз, предназначенная для сбора, обработки, хранения и анализа данных из открытых источников с последующим сопоставлением угроз с моделями объектов критической информационной инфраструктуры.

Система объединяет в одном контуре:

- сбор данных из внешних источников;
- хранение статей, признаков угроз и моделей объектов;
- визуальную работу аналитика через web-интерфейс;
- формирование паспорта объекта КИИ;
- сопоставление угроз с конкретными объектами;
- дальнейшую основу для расчёта риск-метрик.

Проект ориентирован на задачу раннего выявления, структурирования и интерпретации киберугроз применительно к защищаемым объектам.

---

## Проблема, которую решает система

При анализе киберугроз у аналитика обычно возникают несколько типовых проблем:

1. Данные о событиях и угрозах приходят из разных источников и не собраны в одном месте.
2. Источники отличаются по структуре, качеству и полноте.
3. Угроза сама по себе не даёт ответа на вопрос, насколько она актуальна для конкретного объекта.
4. Часто отсутствует единое пространство, где можно одновременно видеть источники данных, реестр угроз и модель защищаемого объекта.
5. Сбор, фильтрация и сопоставление информации нередко выполняются вручную, что увеличивает время анализа и риск пропуска значимых сигналов.

**ALERTA 2.0** решает эту проблему за счёт единой платформы, где аналитик:

- запускает сбор данных;
- видит структурированные записи;
- формирует паспорт объекта;
- получает основу для привязки угроз к объекту;
- использует единое информационное пространство вместо набора разрозненных инструментов.

---

## Цель проекта

Цель проекта — создать прототип интеллектуальной платформы, которая позволяет:

- собирать данные о киберугрозах из открытых источников;
- аккумулировать их в единой базе;
- анализировать и структурировать эти данные;
- формировать модели объектов КИИ;
- использовать модель объекта как основу для дальнейшего сопоставления с моделью угроз;
- готовить базу для расчёта приоритетов реагирования и оценки рисков.

---

## Основная идея системы

Система строится вокруг следующей логики:

1. Внешние источники поставляют данные о событиях, инцидентах, уязвимостях и угрозах.
2. Backend-модуль crawler собирает данные и приводит их к единой структуре.
3. Данные сохраняются в MongoDB.
4. Аналитик работает с ними через web-интерфейс.
5. Для защищаемых объектов создаются отдельные паспорта и модели.
6. На следующем этапе угрозы сопоставляются с конкретными объектами по отрасли, региону, типу объекта, критичности и параметрам воздействия.

Иными словами:

**угроза становится полезной только тогда, когда она связана с конкретным объектом.**

## Функциональные возможности системы

### 1. Сбор данных

Система поддерживает запуск сценариев парсинга через интерфейс.

На текущем этапе в интерфейсе предусмотрены сценарии:

- парсинг сайтов;
- парсинг Telegram;
- парсинг форумов;
- запуск по всем источникам.

Для сценариев предусмотрено окно логов, которое позволяет видеть ход выполнения сбора данных.

### 2. Хранение и просмотр данных

Во frontend предусмотрены разделы для просмотра содержимого базы:

- база данных;
- выявленные угрозы;
- новостные статьи.

Это даёт аналитику возможность просматривать и фильтровать собранный массив информации.

### 3. Формирование модели объекта

В разделе **«Модель объекта»** реализованы:

- вкладка **«Паспорт КИИ»**;
- вкладка **«Список моделей»**.

Паспорт КИИ содержит поля, необходимые для описания объекта:

- наименование объекта;
- тип объекта;
- класс значимости;
- отрасль и подотрасль;
- регион;
- ответственное подразделение;
- бизнес-критичность;
- влияние на конфиденциальность, целостность и доступность;
- параметры экспозиции;
- технологический профиль;
- зрелость защиты;
- рабочие комментарии.

Список моделей отображает уже созданные модели объектов и позволяет:

- просматривать записи;
- фильтровать их;
- искать по ключевым полям;
- удалять модели.

### 4. Подготовка к сопоставлению угроз и объектов

Важная особенность проекта состоит в том, что паспорт объекта строится не как формальная карточка, а как основа для будущих вычислений:

- релевантность угрозы для объекта;
- уровень воздействия на объект;
- оценка экспозиции объекта;
- приоритетность угрозы.

## Архитектура системы

Проект построен по классической клиент-серверной схеме.

### Frontend

Frontend реализован на:

- React;
- React Router;
- Bootstrap / React Bootstrap.

Frontend отвечает за:

- визуализацию данных;
- навигацию по разделам;
- формы ввода;
- запуск сценариев сбора;
- отображение логов;
- просмотр списков записей;
- работу аналитика с моделью объекта.

### Backend

Backend реализован на:

- NestJS;
- Mongoose;
- MongoDB.

Основные backend-модули:

- `AuthModule` — аутентификация и доступ;
- `ArticleModule` — работа со статьями и структурированными записями угроз;
- `CrawlerModule` — запуск сценариев сбора данных;
- `ObjectModule` — хранение и управление моделями объектов;
- `ThreatPredictorModule` — задел под прогнозирование и интеллектуальную интерпретацию угроз.

### База данных

Основное хранилище — **MongoDB**.

Ключевые коллекции:

- `articles` — статьи, новости, записи угроз;
- `objects` — модели и паспорта объектов.

Дополнительно в системе используется:

- **Qdrant** — как основа для векторного хранения и дальнейшей интеллектуальной обработки.

## Логика работы пользователя

Сценарий работы аналитика в системе выглядит следующим образом:

1. Пользователь входит в систему.
2. Открывает панель угроз и разделы содержимого.
3. Запускает сбор данных по выбранным источникам.
4. Анализирует загруженные статьи и признаки угроз.
5. Создаёт паспорт объекта КИИ.
6. Сохраняет модель объекта в базу данных.
7. Переходит в список моделей объектов.
8. Использует модель объекта как основу для дальнейшего анализа и сопоставления с угрозами.

## Что нового и ценного в проекте

### 1. Единый контур

Система объединяет сбор данных, анализ угроз и моделирование объекта в одном интерфейсе.

### 2. Переход от потока новостей к объектно-ориентированному анализу

Проект не ограничивается простым отображением статей и угроз. Он подводит аналитика к более зрелой логике:

- не просто «что происходит в мире угроз»,
- а «что из этого реально важно для моего объекта».

### 3. Основа для риск-ориентированного подхода

Паспорт объекта построен так, чтобы позже использовать его в математической модели сопоставления угроз и объекта.

### 4. Масштабируемость

Архитектура проекта позволяет расширять:

- число источников;
- методы аналитики;
- глубину оценки угроз;
- набор параметров модели объекта;
- методы интеллектуальной классификации.

## Почему проект актуален

Актуальность проекта обусловлена тем, что:

- количество информации о киберугрозах постоянно растёт;
- аналитикам сложно вручную отслеживать и структурировать весь поток;
- организациям необходим переход от общего мониторинга угроз к объектно-ориентированному анализу;
- для КИИ особенно важно понимать не только наличие угрозы, но и её применимость к конкретной инфраструктуре.

Таким образом, **ALERTA 2.0** — это шаг от простого мониторинга к осмысленной аналитической платформе для поддержки принятия решений в области кибербезопасности.
